## 2. 根因分析

### 问题定位过程

#### 第一层现象：密码不匹配
- 使用 `123456` 登录 → ❌ 失败
- 使用 `admin123` 登录 → ✅ 成功

#### 第二层原因：默认密码配置错误

**文件位置**：[server/routes/auth.js:58](file:///home/liuyeming/work/crm/server/routes/auth.js#L58)

```javascript
// 修改前（错误）
if (!user && loginPhone === 'admin') {
  const hashedPassword = await bcrypt.hash('admin123', 10)  // ← 默认密码是 admin123
  user = await User.create({
    phone: 'admin',
    password: hashedPassword,
    ...
  })
}
```

**根因说明**：
1. 首次登录时，代码自动创建admin用户，**默认密码硬编码为 `admin123`**
2. 数据库中存储的是 `admin123` 的hash值
3. 用户尝试用 `123456` 登录 → bcrypt.compare() 返回 false
4. 系统返回"密码错误"

#### 第三层深层原因：环境变量配置错误（部署时）

**关键发现**：
在修复过程中发现PM2启动时使用了**错误的环境变量名**：

| 项目 | 正确值 | 错误值（我设置的）|
|------|--------|------------------|
| **环境变量名** | `MONGODB_URI` | `MONGO_URI` |
| **代码引用** | [server.js:61](file:///home/liuyeming/work/crm/server/server.js#L61) | - |

```javascript
// server.js 第61行
mongoose.connect(process.env.MONGODB_URI)  // ← 期望 MONGODB_URI
```

**影响**：
- PM2使用错误的`MONGO_URI`启动 → MongoDB连接失败
- 即使重置了数据库密码，API服务器也无法读取到新数据
- 导致修复验证阶段出现反复失败的情况

## 3. 修复方案

### 修改1：更新默认密码为 `123456`

**文件**：[auth.js](file:///home/liuyeming/work/crm/server/routes/auth.js)
**位置**：第58行
**改动**：

```diff

+ const hashedPassword = await bcrypt.hash('123456', 10)
```

**理由**：
- 符合用户习惯和预期（简单、常用）
- 与部署脚本中的提示保持一致
- 降低记忆成本

### 修改2：重置现有数据库密码

**操作**：通过Node.js脚本直接更新MongoDB

```javascript
// 查找admin用户并重置密码
const user = await User.findOne({ phone: 'admin' });
user.password = await bcrypt.hash('123456', 10);
await user.save();
```

**执行结果**：
