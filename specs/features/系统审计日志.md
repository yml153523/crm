# 系统审计日志功能需求说明书

**创建日期**: 2026-03-19  
**功能模块**: 系统管理  
**优先级**: 高

---

## 1. 功能概述

### 1.1 背景和价值
为 CRM 系统提供完整的审计日志功能，记录所有用户的关键操作和系统事件，满足安全审计、问题追溯、行为分析等需求。

### 1.2 目标用户
- **系统管理员**：查看审计日志，追踪用户行为，分析系统问题
- **运营人员**：了解用户操作习惯，优化产品体验
- **安全人员**：检测异常行为，保障系统安全

### 1.3 核心场景
1. **用户登录审计**：记录所有登录行为（成功/失败）
2. **数据操作审计**：记录增删改查等关键操作
3. **系统事件审计**：记录系统启动、配置变更等事件
4. **异常行为审计**：记录错误、失败、权限拒绝等事件

## 3. 核心流程

### 3.1 日志记录流程

```
用户操作 → 拦截器/中间件 → 提取操作信息 → 格式化日志 → 写入 MongoDB → 返回响应
```

**详细步骤**：
1. 用户发起操作请求（登录、上传视频、删除数据等）
2. 日志中间件拦截请求
3. 提取操作信息（用户、时间、IP、操作类型、请求参数等）
4. 判断是否需要记录日志
5. 格式化日志数据
6. 异步写入 MongoDB 的 `audit_logs` 集合
7. 返回操作响应（不阻塞主流程）

### 3.2 日志查询流程

```
管理员查询 → 验证权限 → 构建查询条件 → 查询 MongoDB → 返回结果 → 分页展示
```

### 3.3 日志导出流程

```
管理员导出 → 验证权限 → 查询数据 → 生成 CSV/Excel → 下载文件
```

## 4. 边界情况和异常处理

### 4.1 输入边界

**边界 1：日志内容超长**
- **场景**：操作参数或结果内容非常长（超过 1000 字符）
- **处理**：截断至 1000 字符，并标注"内容已截断"

**边界 2：并发日志写入**
- **场景**：大量用户同时操作，产生并发日志写入
- **处理**：使用异步写入，批量插入，避免阻塞主流程

**边界 3：特殊字符**
- **场景**：操作内容包含特殊字符、emoji、HTML 标签等
- **处理**：进行转义处理，防止注入攻击

### 4.2 状态边界

**边界 4：未登录用户操作**
- **场景**：未登录用户尝试访问需要认证的接口
- **处理**：记录日志，标记为"匿名用户"，记录 IP 和设备信息

**边界 5：无权限操作**
- **场景**：用户尝试执行无权限的操作
- **处理**：记录日志，标记为"权限拒绝"，记录详细信息

**边界 6：系统异常**
- **场景**：操作过程中发生系统异常
- **处理**：记录异常日志，包含错误堆栈、请求参数等

### 4.3 业务边界

**边界 7：数据库不可用**
- **场景**：MongoDB 连接失败或不可用
- **处理**：降级到本地文件临时存储，待数据库恢复后同步

**边界 8：日志数据量过大**
- **场景**：日志数据超过 100 万条
- **处理**：提示管理员清理或归档，提供批量删除功能

## 5. 验收标准（Acceptance Criteria）

### AC-001：记录用户登录日志

**Given** 用户尝试登录系统  
**When** 用户提交登录请求（无论成功或失败）  
**Then** 系统应记录以下信息：
- 登录时间（精确到毫秒）
- 用户 ID（如果成功）
- 用户名/手机号
- 登录结果（成功/失败）
- 失败原因（如果失败）
- IP 地址
- 设备信息（浏览器、操作系统）
- User-Agent 原始字符串

**示例**：
```json
{
  "actionType": "LOGIN",
  "userId": "69bb5ba95f5875b7285eaa1f",
  "username": "张三",
  "phone": "13800138000",
  "loginTime": "2026-03-19T10:30:00.000Z",
  "success": true,
  "failReason": null,
  "ipAddress": "192.168.1.100",
  "userAgent": "Mozilla/5.0 (iPhone; CPU iPhone OS 14_0 like Mac OS X)",
  "deviceInfo": {
    "device": "iPhone",
    "os": "iOS 14.0",
    "browser": "Safari"
  }
}
```

### AC-002：记录数据创建操作

**Given** 用户创建新数据（如上传视频、添加会员）  
**When** 创建操作成功完成  
**Then** 系统应记录：
- 操作时间
- 操作用户
- 操作类型（CREATE）
- 数据类型（如 Video、User）
- 数据 ID
- 操作结果
- 关键参数（如视频标题、会员姓名）

---

### AC-003：记录数据修改操作

**Given** 用户修改现有数据  
**When** 修改操作成功完成  
**Then** 系统应记录：
- 操作时间
- 操作用户
- 操作类型（UPDATE）
- 数据类型
- 数据 ID
- 修改的字段（仅记录变更的字段）
- 修改前后的值（可选，敏感字段不记录）

---

### AC-004：记录数据删除操作

**Given** 用户删除数据  
**When** 删除操作成功完成  
**Then** 系统应记录：
- 操作时间
- 操作用户
- 操作类型（DELETE）
- 数据类型
- 数据 ID
- 删除前的关键信息（如标题、名称）

---

### AC-005：记录系统事件

**Given** 系统发生重要事件（如服务重启、配置变更）  
**When** 事件发生  
**Then** 系统应自动记录：
- 事件时间
- 事件类型（SYSTEM_EVENT）
- 事件描述
- 事件级别（INFO/WARNING/ERROR）
- 相关参数

---

### AC-006：记录异常和错误

**Given** 系统发生异常或错误  
**When** 异常被捕获  
**Then** 系统应记录：
- 异常时间
- 异常类型
- 异常消息
- 错误堆栈（简化版）
- 触发异常的用户和操作
- 请求参数

---

### AC-007：按时间范围查询日志

**Given** 管理员需要查询特定时间段的日志  
**When** 管理员设置开始时间和结束时间  
**Then** 系统应：
- 返回该时间段内的所有日志
- 按时间倒序排序（最新的在前）
- 支持分页（每页 20 条）
- 显示总记录数

---

### AC-008：按用户查询日志

**Given** 管理员需要查看特定用户的操作日志  
**When** 管理员选择用户或输入用户 ID  
**Then** 系统应：
- 返回该用户的所有操作日志
- 支持组合时间范围筛选
- 支持分页

---

### AC-009：按操作类型查询日志

**Given** 管理员需要查看特定类型的操作日志  
**When** 管理员选择操作类型（如 LOGIN、CREATE、UPDATE、DELETE）  
**Then** 系统应：
- 返回匹配操作类型的日志
- 支持组合其他筛选条件
- 支持分页

---

### AC-010：关键词搜索日志

**Given** 管理员需要快速定位相关日志  
**When** 管理员输入关键词（如用户名、操作类型、数据 ID）  
**Then** 系统应：
- 在日志的多个字段中搜索（用户名、操作类型、描述等）
- 高亮显示匹配的关键词
- 返回匹配的日志列表
- 支持分页

---

### AC-011：导出日志数据

**Given** 管理员需要导出日志进行离线分析  
**When** 管理员选择导出条件和时间范围  
**Then** 系统应：
- 生成 CSV 或 Excel 文件
- 包含所有匹配的日志记录
- 文件名包含导出时间和条件
- 提供下载链接

---

### AC-012：日志统计面板

**Given** 管理员希望了解日志统计数据  
**When** 管理员访问日志统计页面  
**Then** 系统应显示：
- 今日登录次数
- 今日操作次数（按类型分组）
- 活跃用户 Top 10
- 异常事件统计
- 趋势图表（最近 7 天）

---

### AC-013：日志保留策略

**Given** 日志数据需要长期保存  
**When** 系统定期执行清理任务  
**Then** 系统应：
- 默认保留所有日志（永久保存）
- 支持管理员手动清理指定时间前的日志
- 清理前提示确认
- 记录清理操作本身

---

### AC-014：性能要求

**Given** 日志记录不应影响主流程性能  
**When** 高并发场景下（100+ QPS）  
**Then** 系统应：
- 日志写入延迟 < 50ms（异步）
- 主流程响应时间增加 < 10%
- 不阻塞用户操作

---

### AC-015：安全性要求

**Given** 日志中包含敏感信息  
**When** 记录日志时  
**Then** 系统应：
- 不记录密码明文
- 不记录银行卡号等敏感信息
- 对手机号、身份证号等进行脱敏处理
- 仅管理员可访问日志

## 7. 数据模型（供技术方案参考）

### 7.1 日志集合结构

**集合名称**: `audit_logs`

**字段设计**:
```javascript
{
  _id: ObjectId,              // 日志 ID
  actionType: String,         // 操作类型：LOGIN, CREATE, UPDATE, DELETE, SYSTEM_EVENT, ERROR
  userId: ObjectId,           // 用户 ID（可为空）
  username: String,           // 用户名
  phone: String,              // 手机号（脱敏）
  actionTime: Date,           // 操作时间
  success: Boolean,           // 是否成功
  failReason: String,         // 失败原因
  ipAddress: String,          // IP 地址
  userAgent: String,          // User-Agent 原始字符串
  deviceInfo: {               // 设备信息
    device: String,           // 设备类型：PC, Mobile, Tablet
    os: String,               // 操作系统
    browser: String,          // 浏览器
    version: String           // 浏览器版本
  },
  resourceType: String,       // 资源类型：User, Video, Project, Order 等
  resourceId: ObjectId,       // 资源 ID
  resourceInfo: String,       // 资源描述（如视频标题）
  actionParams: Object,       // 操作参数（脱敏后）
  changes: {                  // 变更内容（仅 UPDATE 操作）
    before: Object,           // 修改前的值
    after: Object             // 修改后的值
  },
  errorMessage: String,       // 错误消息
  errorStack: String,         // 错误堆栈（简化版）
  level: String,              // 日志级别：INFO, WARNING, ERROR
  createdAt: Date             // 创建时间（用于索引）
}
```

**索引设计**:
```javascript
{
  actionTime: -1,                    // 时间倒序索引
  userId: 1,                         // 用户 ID 索引
  actionType: 1,                     // 操作类型索引
  resourceType: 1,                   // 资源类型索引
  { actionTime: -1, actionType: 1 }  // 复合索引
}
```
