### 7.2 分支命名规范

```

```

| 类型 | 用途 | 示例 |
|------|------|------|
| `feat/` | 新功能 | [例如：`feat/article-publish`] |
| `fix/` | Bug 修复 | [例如：`fix/auth-redirect-loop`] |
| `refactor/` | 重构 | [例如：`refactor/supabase-client`] |
| `docs/` | 文档更新 | [例如：`docs/api-spec`] |
| `chore/` | 构建/配置 | [例如：`chore/upgrade-deps`] |

### 7.3 提交信息规范 (Conventional Commits)

所有 Git 提交信息**必须**遵循 [Conventional Commits](https://www.conventionalcommits.org/) 格式：

```
<type>(<scope>): <subject>

[body]

[footer]
```

#### 类型 (type)

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | |
| `fix` | Bug 修复 | |
| `refactor` | 重构（不改变功能） | |
| `style` | 代码格式调整 | |
| `docs` | 文档变更 | |
| `test` | 测试相关 | |
| `chore` | 构建/工具/依赖 | |
| `perf` | 性能优化 | |
| `ci` | CI/CD 配置 | |

#### 作用域 (scope)

> 从项目的 features 模块和通用领域中提取，用表格列出

| scope | 对应范围 |
|-------|---------|
| [模块1] | [范围说明] |
| [模块2] | [范围说明] |
| ... | ... |

#### 提交信息规则
- `subject`：[语言]、[大小写]、[标点]、[长度限制]
- `body`：解释**为什么**，不是**做了什么**
- `footer`：Issue 引用格式

> 至少给出 2 个完整的、贴近本项目业务的提交信息示例

## 13. AI 协作协议

> 本节定义 AI 辅助开发时必须遵守的规则，确保 AI 生成的代码与人工代码保持一致性。

### 13.1 AI 写代码前必读

| 文档 | 目的 |
|------|------|
| [文档路径1] | [读取目的] |
| [文档路径2] | [读取目的] |
| [文档路径3] | [读取目的] |
| [文档路径4] | [读取目的] |
| 相关模块的现有代码 | 理解当前实现，保持一致性 |

### 13.2 AI 写代码的规则

| 规则 | 说明 |
|------|------|
| **读后写** | 修改文件前必须先读取该文件 |
| **最小改动** | 只修改任务要求的部分，不做"顺便优化" |
| **不猜版本** | 安装依赖时使用包管理器获取最新版，不手写版本号 |
| **不引入新依赖** | 除非任务明确要求 |
| **不写无意义注释** | 不写复述代码的注释 |
| **不创建多余文件** | 不主动创建文档或配置文件 |
| **遵循现有模式** | 相同类型的代码必须遵循已有文件的写法 |
| **提交信息规范** | 严格遵循 Conventional Commits |

### 13.3 AI 写完代码后必做

| 步骤 | 操作 |
|------|------|
| **1. 类型检查** | 确认无编译错误 |
| **2. Lint 检查** | 确认无 Lint 错误 |
| **3. 依赖方向** | 确认没有违反目录结构中的依赖方向原则 |
| **4. 环境变量** | 如有新增，同步更新模板文件 |

### 13.4 错题本机制

当 AI 生成的代码出现问题时，在 [错题本文件路径] 中记录：

| 记录项 | 说明 |
|--------|------|
| **错误类型** | 类型错误 / 逻辑错误 / 规范违规 / 依赖错误 |
| **错误描述** | 具体出了什么问题 |
| **根因分析** | 为什么会犯这个错 |
| **正确做法** | 下次应该怎么做 |
