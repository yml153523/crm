## 核心方法论：TDD（测试驱动开发）

本 Skill 强制使用 **Red-Green-Refactor** 循环来完成每一个开发任务：

```
RED（红灯）→ GREEN（绿灯）→ REFACTOR（重构）
  ↑                                    │
  └────────────────────────────────────┘
         下一个任务 / 下一组测试
```

- **RED**：先根据需求和验证标准编写测试，运行确认测试失败
- **GREEN**：编写刚好让测试通过的最少代码
- **REFACTOR**：在测试保护下优化代码结构，确保测试始终通过

**铁律：没有失败的测试，就不写实现代码。**

## 工作流程

### 1. 读取相关文档
**必须按顺序读取以下文档，建立完整的功能上下文：**

#### 1.1 读取需求文档
- 文件路径：`specs/features/{{功能名称}}.md`
- 重点关注：
  - 功能的背景和价值
  - 用户角色和用户故事
  - 核心流程和交互规则
  - 边界情况和异常处理
  - **验收标准（AC）** - 这是测试用例的来源

#### 1.2 读取技术方案文档
- 文件路径：`specs/features/{{功能名称}}_技术方案.md`
- 重点关注：
  - API 接口定义（路径、参数、响应格式）
  - 数据库设计（表结构、字段、索引）
  - 核心逻辑和算法
  - 异常处理方案
  - 技术难点和解决方案

#### 1.3 读取任务规划文档
- 文件路径：`specs/features/{{功能名称}}_任务规划.md`
- 重点关注：
  - 当前任务的详细信息
  - 提取每个任务的：
    - 任务编号（如 Task-01）
    - 任务标题和说明
    - 涉及文件
    - 测试文件（**RED 阶段创建测试文件时必须使用此路径**）
    - 对应的验收标准
    - 依赖关系
    - 验证标准（**这是编写测试的直接依据**）

### 2. 前置检查
在开始编码前，必须检查：
- [ ] **依赖任务**：前序任务产出的文件/模块是否存在
- [ ] **项目环境**：项目能否正常启动，依赖库是否安装
- [ ] **测试环境**：测试框架已配置，能正常运行测试
- [ ] **上下文理解**：确认已理解需求文档和任务规划的关键内容
- [ ] **开发服务器**：如果当前任务包含浏览器端测试场景（UI 变化或页面可感知的行为变更），必须检查开发服务器（`http://localhost:3300`）是否已在运行。**如果未运行，必须自行启动开发服务器后再继续执行。**

**如果前置条件不满足，必须停止并在输出报告中说明原因。**

### 3. 执行开发任务（TDD 循环）
按照任务顺序，逐个完成。每个任务严格遵循 **Red-Green-Refactor** 循环：

#### 3.1 RED - 编写失败的测试

**在编写任何实现代码之前，先编写测试。**

1. **分析测试需求**
   - 根据任务的**验证标准**和**对应的验收标准（AC）**，确定需要哪些测试用例
   - 测试必须覆盖：
     - 正常情况（Happy Path）
     - 边界情况（Edge Cases）
     - 异常情况（Error Cases）

2. **编写测试文件**
   - 测试类型根据任务性质选择：
     - 数据层任务 -> 单元测试（测试 CRUD 操作）
     - API 任务 -> 集成测试（测试接口响应）
     - 组件任务 -> 组件测试（测试渲染和交互）
     - 工具函数 -> 单元测试（测试输入输出）
   - 测试文件命名规范：
     - `tests/unit/[模块名].test.js`
     - `tests/integration/[功能名].test.js`
     - `tests/components/[组件名].test.js`
   - 测试描述要清晰（describe/it 语句要说明测试内容）
   - 每个测试只验证一个行为点

3. **运行测试，确认失败（RED）**
   - 运行刚编写的测试
   - **测试必须失败**（因为还没有实现代码）
   - 确认失败原因是"功能未实现"，而不是测试本身有 bug
   - 在输出报告中记录测试失败的结果

**如果测试在没有实现代码的情况下就通过了，说明：**
- 测试写得有问题（没有真正验证功能），需要修正测试
- 或者功能已经被其他代码实现了，需要重新评估任务

#### 3.2 GREEN - 编写最少的实现代码

**目标：用最少的代码让测试通过，不多不少。**

1. **编写实现代码**
   - **严格遵守需求文档**：确保实现符合用户故事和验收标准
   - **严格遵守项目规范**：遵守技术栈和开发规范
   - **优先复用代码**：复用项目中已有的代码、组件、工具函数
   - **克制过度实现**：只写让当前测试通过所需的代码，不要提前实现后续任务的功能

2. **运行测试，确认通过（GREEN）**
   - 运行所有相关测试（包括之前任务的测试，防止回归）
   - **所有测试必须通过**
   - 如果测试失败，修复实现代码（不是修改测试）直到通过
   - 在输出报告中记录测试通过的结果

#### 3.3 REFACTOR - 重构优化

**在测试保护下，改善代码质量。**

1. **代码审视**
   - 检查是否有重复代码可以提取
   - 检查命名是否清晰准确
   - 检查代码结构是否合理
   - 检查是否符合项目规范

2. **执行重构**（如果需要）
   - 提取公共方法/组件
   - 改善命名和代码组织
   - 优化性能（如有明显问题）
   - 添加必要的注释

3. **运行测试，确认仍然通过**
   - 每次重构后都要运行测试
   - **测试必须始终保持通过状态**
   - 如果测试失败，回退重构改动

**重要：如果跳过了 RED 阶段（没有先写测试），任务不算完成。**

### 4. 自动化验收测试

TDD 循环完成后，必须通过验收测试验证任务成果在真实环境中的正确性。

**测试依据**：
- 当前任务的**验证标准**（见上方"当前任务"章节）
- 当前任务**对应的验收标准（AC）**（来源：`specs/features/{{功能名称}}.md` 中的验收标准章节）
- **需求文档**的完整上下文（`specs/features/{{功能名称}}.md`）

#### 4.1 浏览器端测试（优先）

**适用场景**：当前任务有 UI 变化或页面可感知的行为变更。

使用 **Playwright MCP** 对 `http://localhost:3300` 进行端到端验证。

按照 AC 描述的用户操作路径执行交互，验证页面状态、显示内容、交互反馈是否符合预期，并在关键验证点截图留档。

**强制停止规则**：如果存在浏览器端测试场景，但由于任何原因（开发服务器无法启动、环境变量缺失、浏览器启动失败、页面加载错误等）无法执行浏览器端测试，**必须立即停止当前任务**，在报告中详细说明无法执行的原因，不得跳过浏览器端测试继续完成任务。

#### 4.2 单元测试（兜底）

**适用场景**：纯后端逻辑、工具函数、数据处理等无法通过浏览器验证的任务。

此类任务在 TDD 阶段已编写单元测试，确认 TDD 测试全部通过即视为验收通过。

#### 4.3 数据验证

**适用场景**：涉及数据库变更的任务（如创建、修改、删除数据）。

使用 **Supabase MCP**（`execute_sql`）查询数据库，验证数据是否符合预期：

- 数据是否正确写入/更新/删除
- 字段值、数据类型、约束是否符合设计
- 关联数据的一致性

**注意**：数据验证可与浏览器端测试或单元测试组合使用，不互斥。

### 5. 修复与重测循环

若验收测试未通过，进入修复循环：

```
验收测试失败 → 分析失败原因 → 修复代码 → 重跑 TDD 测试（防回归）→ 重跑验收测试 → 仍失败则继续循环
```

**规则：**
- **最大重试次数：3 次**。超出仍未通过，标记任务为失败，在报告中详细说明失败原因和已尝试的修复措施
- 每次修复后，必须同时重跑 TDD 阶段的测试，确保修复未引入回归
- 每次修复需在报告中记录：问题描述、修复措施、重测结果

### 6. 文档同步

修复过程中，如果实现逻辑与现有功能文档产生偏差，必须同步更新相关文档：

- 检查 `specs/features/` 下的需求文档和任务规划文档
- 如果代码行为与文档描述不一致，更新文档使其与实际实现对齐
- 在报告中列出所有文档变更

**如果未经历修复且代码与文档一致，跳过此步骤。**

## 关键规则

### 规则 1：严格遵守任务规划
- 不要做任务要求之外的事情
- 不要提前实现后续任务的功能
- 严格按照技术方案中的设计来实现

### 规则 2：代码复用优先
在编写新代码前，必须：
1. 检查项目中是否有类似的实现
2. 检查是否有可复用的组件、工具函数
3. 遵循项目现有的代码模式和风格

### 规则 3：TDD 铁律
**每个任务必须严格遵循 Red-Green-Refactor 循环，不可跳步：**

1. **不写没有测试的实现代码** — 每一行实现代码都由一个失败的测试驱动出来
2. **不写超出测试范围的代码** — 只写让当前测试通过所需的最少代码
3. **RED 必须真红** — 测试必须因为"功能未实现"而失败，不是因为语法错误或环境问题
4. **GREEN 必须刚好绿** — 通过测试后立即停止编码，进入重构阶段
5. **REFACTOR 不改行为** — 重构只改结构不改功能，测试必须始终通过
6. **测试文件与源文件对应** — 如 `user.js` -> `user.test.js`
7. **每个测试只验证一个行为** — 保持测试的原子性和可读性

**违反检查清单（出现以下情况必须停止并修正）：**
- 写了实现代码但没有先写测试 -> 回退，先补测试
- 测试在没有实现的情况下就通过了 -> 测试无效，重写测试
- 重构后测试失败了 -> 回退重构改动

### 规则 4：文档同步
如果代码修改影响了文档，必须：
- 更新相关的 API 文档
- 更新 README（如果需要）
- 在输出报告中说明文档变更

### 规则 5：透明沟通
- **在 RED 阶段记录测试代码和失败结果**
- **在 GREEN 阶段记录实现代码和通过结果**
- **在 REFACTOR 阶段说明重构内容和测试结果**
- 遇到问题时，在输出报告中详细说明

### 规则 6：不更新任务规划文档
**禁止修改任务规划文档的勾选状态。** 任务状态由主 Agent 统一管理。
